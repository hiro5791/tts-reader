# 読み上げアプリ 追加実装 指示書（フェーズ3：標準機能セット）

> この文書は、Claude Code に渡すための「指示書」です。
> 既存アプリ（Multi Voice Studio）に、標準レベルの機能を追加します。
> **フェーズ1（最小版）・フェーズ2（声選択・速度）が動いている前提**です。
> Claude Code には「この指示書の上から、ステップごとに区切って、動作を確認しながら進めてください」と頼んでください。

## 0. この文書について（前提）

今回足すのは、次の5つです（すべて `読み上げアプリ_機能一覧.md` の **標準レベル**）。

1. **生成中はテキストを編集できないようにする**（誤操作・ハイライトずれの防止）
2. **ファイル読み込み**（txt / PDF / Word → 入力欄へ）
3. **Irodori の長文分割（チャンク）**（Qwen は分割しない）
4. **音量・ピッチの調整**
5. **読み上げ位置のハイライト**

守ってほしい前提（フェーズ1・2から継続）：

- **アダプタ方式を維持する。** UI はエンジンの違いを知らない。共通の窓口 `synthesize(...)` 越しに頼むだけ。
- **重い処理は別スレッド。** 生成中も UI が固まらないようにする（既存の仕組みに乗せる）。
- **UI は CustomTkinter のデスクトップアプリ。** ブラウザ（Web/Gradio）ではない。
- **新しいエンジンは追加しない**ので、今回ライセンス確認は不要（Qwen3 / Irodori のまま）。

## 1. 実装の順番（おすすめ）

「小さく安全なものから、複雑なものへ」の順です。各機能は独立しているので、1つ足すごとに動作確認できます。

1. **生成中のテキスト編集ロック** … いちばん小さい。先に入れておくと後の機能が安全になる
2. **ファイル読み込み** … UI 内で完結。他と干渉しない
3. **Irodori 長文分割** … バックエンド（Irodori アダプタの中だけ）
4. **音量・ピッチ調整** … バックエンド後処理＋スライダー2本
5. **読み上げ位置のハイライト** … 再生と連動するので、いちばん複雑。最後にやる

## 2. 共通の窓口の拡張（先に決めておくこと）

今回の機能のうち「速度・音量・ピッチ」は、共通の窓口 `synthesize` の引数を増やして実現します。
フェーズ2までで `synthesize(text, engine, voice, speed)` まで来ているはずなので、こう広げます：

```python
synthesize(
    text: str,
    engine: str,                 # "qwen3" | "irodori"
    voice: str | None = None,
    speed: float = 1.0,          # 1.0 = 等倍（フェーズ2で実装済み）
    volume: float = 1.0,         # 1.0 = 等倍（今回追加）
    pitch: float = 0.0,          # 0 = 変化なし。半音単位（今回追加）
    progress_callback=None,      # 分割生成の進捗通知用（今回追加）
) -> 音声（wavパス または 音声バイト列）
```

### 後処理（DSP）を1か所にまとめる

**重要な設計判断：** 両エンジンとも「音量つまみ」「ピッチつまみ」を持っていません（確認済み）。
なので音量・ピッチは**エンジンに頼らず、生成後の音声を加工する「後処理」で実現**します。
これは予測可能で、どちらのエンジンでも同じように効くので好都合です。

後処理は、各アダプタにバラバラに書かず、**共通の窓口の中に「後処理ユーティリティ」を1つ作って集約**してください。
処理の順番は次の通り：

```
[ エンジンが生(なま)の音声を作る ]
        ↓
   ① ピッチ変更（pitch != 0 のとき）   … 長さは変えずに音の高さだけ変える
        ↓
   ② 速度変更（speed != 1.0 のとき）   … 高さは変えずに速さだけ変える（フェーズ2の後処理をここへ寄せてよい）
        ↓
   ③ 音量変更（volume != 1.0 のとき）   … 最後にゲインをかける
        ↓
   [ 完成した音声を返す ]
```

- ①ピッチと②速度は**別物**です。ピッチ＝長さ保持のピッチシフト、速度＝高さ保持のタイムストレッチ。
  どちらも `librosa`（`librosa.effects.pitch_shift` / `librosa.effects.time_stretch`）で揃えられるので、依存が増えません。
  品質をさらに上げたいときは `pyrubberband` も候補（rubberband のインストールが要る点だけ注意）。
- フェーズ2で「エンジン側に速度パラメータがあればそれを使う」と書きましたが、無ければこの後処理に集約してOKです。

> こうしておくと、Irodori アダプタ（分割あり）も Qwen アダプタ（分割なし）も、
> 「生の音声を返す」ことだけに集中でき、音量・ピッチ・速度は共通層が面倒を見ます。

## 3. 機能①：生成中はテキストを編集できないようにする

### 目的

生成している最中にテキストを書き換えられると、できあがる音声と画面の文章がズレます。
特に後で入れる**ハイライト**は「生成したときの文章」を基準にするので、途中編集を禁止しておくのが安全です。

### 実装方針（🟦 アプリ側）

既存の「別スレッドで生成」する流れに、ロックの ON/OFF を足すだけです。

- **生成開始時**（スレッドを起動する直前）：
  - テキスト入力欄を編集不可にする … CustomTkinter なら `textbox.configure(state="disabled")`
  - 「生成」ボタンを無効化する（二重押し防止）
  - （任意）エンジン・声・速度などの操作も無効化してよい
  - ステータスに「生成中…」を出す（既存の左下ステータス行を流用）
- **生成完了時 / エラー時**（メインスレッドに戻ってくるコールバックの中）：
  - 上記をすべて元に戻す（`state="normal"` 等）

> 注意：UI 部品の状態変更は**必ずメインスレッドで**行う。別スレッドから直接触らない。
> 既存の「終わったらメインに結果を返す」仕組みの中で戻すこと。

## 4. 機能②：ファイル読み込み（txt / PDF / Word）

### 目的

ファイルからテキストを取り出して入力欄に流し込む。エンジンとは無関係の、純粋なアプリ側機能です。

### 実装方針（🟦 アプリ側）

1. UI に「**ファイルを開く**」ボタンを1つ追加（※ `.mvsvoice` 用の「ファイルを選択」とは別物。ラベルで区別する）。
2. ボタンで `filedialog`（tkinter標準）を開き、拡張子で処理を振り分ける：

| 種類 | 推奨ライブラリ | メモ |
|---|---|---|
| `.txt` | 標準（`open`） | まず UTF-8 で読み、失敗したら **cp932（Shift-JIS）** で再試行。`chardet` を使ってもよい |
| `.pdf` | `pypdf`（軽い） | レイアウトが崩れる/表が多い場合は `pdfplumber` に切り替え |
| `.docx` | `python-docx` | 段落（paragraphs）を改行で連結する |

3. 取り出したテキストを**入力欄に流し込む**。既存テキストは「置き換え」を基本に（追記したいかは後で足せる）。

### 注意点

- **抽出は完璧ではない。** PDF の段組み・表・ヘッダーフッターは乱れることがある。
  あくまで「下書きを流し込む」もので、**ユーザーが入力欄で直してから生成する**前提でよい。
- 旧 `.doc`（Word97-2003）は `python-docx` では読めない。今回は `.docx` のみ対応でよい（必要になったら拡張）。
- 文字数が多い場合は、機能③の分割（Irodori）と機能⑤のハイライトが効いてくる。

## 5. 機能③：Irodori の長文分割（チャンク）

### 目的と方針（重要）

調べた結果、**エンジンによって扱いが正反対**でした。ここが今回の肝です。

- **Irodori-TTS**：一度に生成できる長さに**約30秒の上限**があり、超えると後半が崩れる/無音になる。→ **分割が必須**。
- **Qwen3-TTS**：10分超の連続生成でも崩れない設計。むしろ分割の繋ぎ目で品質が落ちうる。→ **分割しない**。

なので **「Irodori アダプタの中だけ」に分割を閉じ込め、Qwen アダプタは素通し**にします。
共通の窓口から見ると、どちらも「長文でも1回で音声が返ってくる」ように見えます（アダプタ方式の利点）。

### 分割ルール（Irodori アダプタ内）

1. テキストを **「。」「！」「？」** で文に区切る。
2. 文を順に足していき、**累計が100文字以上になったら、そこで1チャンク確定**（＝文の途中では切らない。「100文字を超えた最初の文末で切る」）。
3. **保険**：1つの文だけで 120〜150文字を超える場合は、その文を **「、」** でさらに分割する（読点だらけの長文対策）。
4. 各チャンクを順にエンジンへ渡して生成し、**チャンクの間に 0.2〜0.3秒の無音**を挟んで1本に連結する。
5. 連結する音声の**サンプルレートを揃える**（同一エンジンなら通常そろう）。

```
chunk = ""
chunks = []
for 文 in split_by_。！？(text):
    chunk += 文
    if len(chunk) >= 100:
        chunks.append(chunk)
        chunk = ""
if chunk:
    chunks.append(chunk)
# （各 chunk が長すぎる場合は「、」で再分割する保険を入れる）
```

### 進捗通知

- 分割生成中は `progress_callback(i, n)`（例：3/8）を呼び、UI が**左下ステータスに「生成中…（3/8）」**を出せるようにする。
  - コールバックは別スレッドから呼ばれるので、UI 反映はメインスレッドへ（`after` 等で）。

### 注意点

- **閾値（100文字）・無音の長さ（0.2〜0.3秒）は、実機で読ませて崩れ始める所を見て微調整**してよい。
  まずは「100文字 / 0.25秒無音」で動かし、聞いて決める。
- Qwen アダプタには**この分割を入れない**。そのまま渡す。

## 6. 機能④：音量・ピッチの調整

### 目的

ユーザーが音量とピッチ（声の高さ）を変えられるようにする。

### 実装方針（🟩 両方 → 実際は後処理）

セクション2で決めた通り、**共通層の後処理**で実現します（エンジンには音量/ピッチのつまみが無いため）。

- **音量（volume）**：線形ゲインを掛ける（`samples * gain`）。最後に必ず **[-1.0, 1.0] にクリップ**して歪み（クリッピング）を防ぐ。
  - UI は **%表示**が分かりやすい（100% = 等倍 = `1.0`）。範囲は例：0〜200%。
- **ピッチ（pitch）**：**半音単位**で `librosa.effects.pitch_shift` を使う（長さは変えない）。
  - UI は **−12〜+12 半音、初期値0**。0 のときは処理をスキップ（無駄な劣化を避ける）。

### アダプタ・UI への追加

- 共通の窓口に `volume` `pitch` を渡す（セクション2の署名）。各アダプタは生の音声を返すだけ。
- UI：既存の **「速度」スライダーの並び**に、**「音量」スライダー**と**「ピッチ」スライダー**を追加する。
  - 値の現在地（例：`120%` / `+2半音`）をスライダー右に表示すると分かりやすい（速度の「1.0倍」表示と同じ作り）。

### 注意点

- 後処理の**順番はピッチ→速度→音量**（セクション2）。音量は最後。
- ピッチを大きく動かすと声質が不自然になりやすい。まずは ±数半音で試す。

## 7. 機能⑤：読み上げ位置のハイライト

### 目的

いま読んでいる箇所を、入力欄の文章上で光らせる。長文を目で追いやすくする機能。

### 実装方針（🟦 アプリ側）— まずは「文単位」で

単語レベルの正確なハイライトは、各単語の時刻情報（強制アラインメント）が要って重い。
**MVP は「文単位」**で、再生時間を文字数で按分する**シンプルな方式**にします。両エンジンで同じように動きます。

1. **文に分割**：表示中のテキストを「。」「！」「？」で文に分け、**各文の文字インデックス（開始・終了位置）を覚えておく**。
   （Irodori の分割ロジックとは別物。ハイライトは「文単位」で持つ。）
2. **時間配分を見積もる**：完成した音声の総再生時間 `D` を、各文の**文字数の比**で按分する。
   （文 i の長さ ≒ `D × (文 i の文字数 / 全文字数)`）
3. **タイムライン化**：文 i は `t_i 〜 t_{i+1}` に再生される、という時間割を作る。
4. **再生中に追従**：再生開始後、**50〜100msごとに UI タイマー**（CustomTkinter の `after`）で**現在の再生位置（経過秒）**を取り、
   その時刻が含まれる文を探して、その文だけ**背景色を付ける**（前の文の色は消す）。
5. **停止/一時停止**でハイライトを止める。**テキストが編集されたらハイライトを消す**（機能①のロックと連動）。

### 再生位置の取り方

- 再生位置を返せるプレーヤーを使うのが確実（例：`pygame.mixer` の再生位置取得、`sounddevice` でフレーム数を数える等）。
  既存の再生機能が位置を返せないなら、**再生開始時刻からの経過時間**で代用（一時停止がある場合は経過を加算管理）。
- CustomTkinter の `CTkTextbox` は内部の tkinter Text のタグ機能（`tag_config` / `tag_add` / `tag_remove`）が使えるので、
  それで文の範囲に背景色タグを付け外しする。

### 注意点と将来の改善

- **文字数按分なので多少ずれる**（漢字とかなで読む長さが違う、句読点で間が入る等）。文単位なら実用上は許容範囲。
- もっと正確にしたくなったら：Irodori は**実チャンクの再生時間**が分かるのでそれを使う、
  あるいは強制アラインメントで**単語レベル**にする、という改善が後で可能（今回はやらない）。

## 8. 今の画面（Multi Voice Studio）への足し方まとめ

現在の画面を基準に、どこへ何を足すかの一覧です。レイアウトを大きく崩さずに収まります。

| 機能 | 置き場所 | 部品 |
|---|---|---|
| ① 生成中ロック | 既存のテキスト入力欄／生成ボタン | 部品追加なし（生成中に `disabled` にするだけ） |
| ② ファイル読み込み | テキスト入力欄の上 or すぐ近く | 「ファイルを開く」ボタン1つ（`.mvsvoice` の「ファイルを選択」と混同しないラベルに） |
| ③ Irodori分割 | 画面変更なし（裏側） | 進捗だけ**左下ステータス**に「生成中…（3/8）」 |
| ④ 音量・ピッチ | 既存「速度」スライダーの並び | 「音量」スライダー＋「ピッチ」スライダー（各右に現在値表示） |
| ⑤ ハイライト | 既存テキスト入力欄 | 部品追加なし（文に背景色タグを付け外し） |

波形表示・「声を保存する」「音声を保存する」まわりは、今回はそのままで大丈夫です。
