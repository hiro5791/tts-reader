# 読み上げアプリ 設計書（統合版）

> この文書は、Claude Code に渡すための「指示書」です。
> アプリ全体を **フェーズ1 → フェーズ2 → フェーズ3 → フェーズ4** の順で作ります。
> Claude Code には「この設計書のフェーズ1から、ステップごとに区切って、動作を確認しながら進めてください」と頼んでください。
> すでに動いているフェーズがある場合は、未実装のフェーズから続けてください。

---

## 0. このアプリは何か（ゴール）

自分のPCで動く、日本語テキストの読み上げ **Windowsデスクトップアプリ**（名称：Multi Voice Studio）。

テキストを入力すると音声で読み上げ、声・速度・音量・ピッチを選べる。ファイルから読み込め、長文も扱え、
言葉で声を作ったり喋り方を指定したりできる——というのが最終像。
ただし一度に全部は作らず、「動く最小の核」から機能を足していく。

- **フェーズ1（最小版）**：テキスト→音声→再生。Qwen3 と Irodori を切り替えて聞き比べる。
- **フェーズ2（必須機能）**：話者（声）の選択／読み上げ速度の調整。
- **フェーズ3（標準機能）**：生成中ロック／ファイル読み込み／Irodori長文分割／音量・ピッチ／ハイライト／進捗表示／設定保存。
- **フェーズ4（声を作る・喋り方）**：VoiceDesign（言葉で声を作る）／感情・スタイル指定。

---

## 1. 使うエンジン（重要：ライセンス）

- **Qwen3-TTS** … Apache 2.0（商用利用OK）。0.6B または 1.7B モデル。多言語対応（日本語含む）。
- **Irodori-TTS** … 日本語特化。コードは MIT。
  - **日本語のみ対応**。**漢字の読みが弱め**で、難しい漢字は誤読することがある。

どちらも商用でクリーンに使えることが採用理由。
**ライセンスは「モデルの重み」の条件がコードと別管理**のことがあるため、
商用化の前に各モデルカードの最新ライセンスを必ず確認すること。

### エンジンの性質（設計に直結する重要な違い）

| | Qwen3-TTS | Irodori-TTS |
|---|---|---|
| 対応言語 | 10言語（日本語含む） | 日本語のみ |
| プリセット話者 | **あり**（例：おの あんな） | **無い**（指定しないと声がランダムに変わる） |
| 長文 | **10分超でも安定**（分割不要） | **約30秒で頭打ち**（分割が必須） |
| 声を作る(VoiceDesign) | あり（`instruct`） | あり（`caption`／むしろ得意） |
| クローン | あり（約3秒〜） | あり（参照音声） |
| 感情・スタイル | 別引数 `instruct`（声と分離可） | caption に同居 ＋ 本文の絵文字（分離不可） |
| 漢字の読み | 比較的安定 | 弱い |

> この「プリセットの有無」「長文の扱い」「感情指定の作法」の3点が、後のフェーズでUIの出し分けに効いてくる。

## 2. 動作環境

- 開発機：Windows または Linux の、**NVIDIA GPU 搭載PC**。GPU前提（CPUのみは非常に遅い）。

## 3. 全体の設計方針（アダプタ方式）

最初から「アプリ本体（UI）」と「エンジン部分」を分けて作る。**この構造は全フェーズで維持する。**

```
[ UI（テキスト入力＋エンジン選択＋各種設定＋再生） ]
        ↓ 文章・エンジン名・声・速度・音量・ピッチ・モード等を渡す
[ TTSアダプタ（共通の窓口 synthesize / 後処理DSP） ]
        ↓ エンジンごとに振り分け
   ┌──────────────┬──────────────┐
[ Qwen3 アダプタ ]      [ Irodori アダプタ ]
```

- **UI** はエンジンの細かい違いを知らない。アダプタに「この文章を、このエンジンで」と頼むだけ。
- **TTSアダプタ** が共通の窓口。`synthesize(...)` という1つの関数を持つ。
- **音量・ピッチ・速度などの後処理（DSP）は共通層に集約**。各エンジンアダプタは「生の音声を返す」ことに集中する。
- **Irodoriの長文分割は Irodori アダプタの中だけに閉じ込める**（共通層からは「長文でも1回で返る」ように見える）。

### 共通の窓口（最終形の署名）

フェーズが進むにつれ引数が増える。最終的にはこの形（実装準拠）：

```python
synthesize(
    text: str,
    engine: str,                 # "qwen3" | "irodori"
    voice: str | None = None,    # フェーズ2（プリセットの声ID。None なら既定）
    speed: float = 1.0,          # フェーズ2（1.0=等倍）
    volume: float = 1.0,         # フェーズ3（1.0=等倍）
    pitch: float = 0.0,          # フェーズ3（0=変化なし。半音単位）
    mode: str = "preset",        # フェーズ4: "preset" | "voice_design"
    voice_description: str | None = None,  # フェーズ4: 声を作る説明文（Qwen=instruct / Irodori=caption）
    style: str | None = None,    # フェーズ4: 喋り方（Qwen=instruct。Irodoriは未使用）
    language: str | None = None, # 音声言語（None なら各エンジンの既定言語）
    progress_callback=None,      # フェーズ3: 分割生成の進捗通知 callback(i, n)
    cancel_event=None,           # 生成の中断用（停止ボタンから渡す threading.Event）
) -> str  # 生成した wav ファイルのパス
```

- **「保存した声（クローン）」は別の窓口** `synthesize_clone(...)` を使う（`mode="saved"` ではない）：

```python
synthesize_clone(
    engine, text, ref_wav, ref_text=None, language=None,
    speed=1.0, volume=1.0, pitch=0.0,
    style=None,                  # UIからは渡るが現状クローンには未適用（後述）
    progress_callback=None, cancel_event=None,
) -> str
```

# ■ フェーズ1：最小版

## 1-1. ゴール

- テキスト入力欄に文章を貼り付ける
- ボタンを押すと、その文章を音声で読み上げて再生する
- 読み上げエンジンを **Qwen3-TTS** と **Irodori-TTS** から切り替えて聞き比べられる

これ以外（声の選択・速度・ファイル読み込み・オリジナルの声・誤読補正など）はフェーズ1には入れない。

## 1-2. 共通の窓口（インターフェース）

フェーズ1で必要なのは実質これ1つ：

- `synthesize(text, engine) -> 音声`（`engine` は `"qwen3"` か `"irodori"`）

声・速度などのオプションは固定でよい（各エンジンのデフォルト＋日本語のプリセット話者）。
※ 後のフェーズで引数を足すので、拡張しやすい形にしておくこと。

## 1-3. UI（Windowsデスクトップアプリ）

- **ブラウザ画面（Web/Gradio）ではなく、Windows上で動くデスクトップGUIにする。**
- 実装は **CustomTkinter** を基本とする（Python製・今風の見た目・配布しやすい）。
- 画面に必要なもの：
  1. テキスト入力欄（複数行）
  2. エンジン選択（Qwen3 / Irodori）
  3. 「読み上げる（生成）」ボタン
  4. 音声の再生（再生／停止）
  5. 処理中の表示（「生成中…」。初回はモデル読み込みで時間がかかる）
- **重い処理は別スレッドで動かす**：生成中もUIが固まらないよう、生成は裏のスレッドで実行し、
  終わったらメインのUIに結果を返す。

## 1-4. エンジンの呼び出し方（参考）

最新の正確な使い方は各リポジトリのREADMEを確認すること。

### Qwen3-TTS

- パッケージ：`pip install -U qwen-tts soundfile` ／ 公式：https://github.com/QwenLM/Qwen3-TTS
- 日本語の女性話者 `Ono_Anna` の例：

```python
import torch, soundfile as sf
from qwen_tts import Qwen3TTSModel

model = Qwen3TTSModel.from_pretrained(
    "Qwen/Qwen3-TTS-12Hz-1.7B-CustomVoice",
    device_map="cuda:0",
    dtype=torch.float16,
    attn_implementation="sdpa",    # Windows等 flash_attention_2 が使えない環境向け
)
wavs, sr = model.generate_custom_voice(
    text="ここに読み上げる日本語",
    language="Japanese",
    speaker="Ono_Anna",
)
sf.write("out_qwen.wav", wavs[0], sr)
```

### Irodori-TTS

- 公式：https://github.com/Aratako/Irodori-TTS ／ OpenAI互換サーバー：https://github.com/Aratako/Irodori-TTS-Server
- CLI例（参照音声なし）：

```bash
python infer.py \
  --hf-checkpoint Aratako/Irodori-TTS-500M-v3 \
  --text "ここに読み上げる日本語" \
  --no-ref \
  --output-wav out_irodori.wav
```

- Irodori アダプタは「`infer.py` を呼ぶ」か「`Irodori-TTS-Server` をHTTPで叩く」か、やりやすい方でよい。

## 1-5. 作る順番（フェーズ1）

1. **準備**：作業フォルダ・Python環境・GPU確認。
2. **Qwen3 だけ先に**：Qwen3 アダプタで固定テキストを wav 出力→再生。
3. **UIをつなぐ**：CustomTkinter で 入力→ボタン→Qwen3で再生（生成は別スレッド）。
4. **Irodori を足す**：エンジン選択で切り替え。
5. **聞き比べ確認**。

# ■ フェーズ2：必須機能（声選択・速度）

## 2-1. 話者（声）の選択

- **Qwen3（CustomVoice）**：`generate_custom_voice(..., speaker=話者名)` の `speaker` を切り替える。
  日本語話者の一覧は最新のモデルカード／READMEで確認し、UIのプルダウンに並べる。
- **Irodori**：プリセットの概念が薄い。デフォルト声（`--no-ref`）を1つ用意し、
  余裕があれば許諾済み参照音声を「声プリセット」として登録できるようにする。
- 共通窓口に `voice` を足す。`list_voices(engine)` でエンジン別の声一覧を返し、
  エンジンを切り替えたら声のプルダウンも入れ替わるようにする。

## 2-2. 読み上げ速度の調整

- 各エンジンに速度パラメータがあれば使う。無ければ**後処理のタイムストレッチ**（ピッチ保持）で実現。
  - 例：`librosa.effects.time_stretch`、`pyrubberband` 等。
  - 単純なリサンプリングは声が高く/低くなるので不可。
- 共通窓口に `speed`（1.0=等倍）を足す。UIはスライダー（0.5〜2.0、初期1.0）。

> ※ フェーズ3で音量・ピッチを後処理に集約するので、速度の後処理もそこへ寄せてよい。

# ■ フェーズ3：標準機能セット

足すのは次の **7つ**（順番は「小さく安全 → 複雑」）。各機能は独立しているので1つずつ確認できる。

1. 生成中のテキスト編集ロック
2. ファイル読み込み（txt / PDF / Word）
3. Irodori の長文分割（チャンク）— Qwenは分割しない
4. 音量・ピッチの調整
5. 読み上げ位置のハイライト
6. 進捗表示（分割時のみ）
7. 設定の保存（最小限）

## 3-0. 後処理（DSP）を1か所に集約

両エンジンとも音量・ピッチのつまみを持っていないので、**生成後の後処理**で実現する。
各アダプタにバラバラに書かず、**共通層の後処理ユーティリティに集約**する。処理順は：

```
[ エンジンが生の音声を作る ]
   → ① ピッチ変更（pitch != 0）   長さは変えず高さだけ
   → ② 速度変更（speed != 1.0）   高さは変えず速さだけ（フェーズ2の後処理をここへ寄せる）
   → ③ 音量変更（volume != 1.0）  最後にゲイン
   → [ 完成した音声を返す ]
```

- ピッチ＝`librosa.effects.pitch_shift`、速度＝`librosa.effects.time_stretch`（同じライブラリ系統で依存が増えない）。
  品質を上げたいなら `pyrubberband` も候補（rubberband のインストールが要る）。

## 3-1. 生成中のテキスト編集ロック（🟦 アプリ側）

生成中に文章を書き換えられると、音声と画面がズレる（特にハイライトの基準が壊れる）。

- **生成開始時**（スレッド起動の直前）：テキスト欄を `state="disabled"`、生成ボタン無効化、（任意で）他操作も無効化、「生成中…」表示。
- **完了/エラー時**（メインスレッドのコールバック内）：すべて元に戻す。
- UI部品の状態変更は**必ずメインスレッド**で行う。

## 3-2. ファイル読み込み（txt / PDF / Word）（🟦 アプリ側）

- 「**ファイルを開く**」ボタンを追加（`.mvsvoice` 用の「ファイルを選択」とは別物。ラベルで区別）。
- `filedialog` で開き、拡張子で振り分け：

| 種類 | 推奨ライブラリ | メモ |
|---|---|---|
| `.txt` | 標準（`open`） | UTF-8で試し、失敗したら cp932（Shift-JIS）で再試行（`chardet` も可） |
| `.pdf` | `pypdf` | レイアウト崩れ・表が多いときは `pdfplumber` |
| `.docx` | `python-docx` | 段落を改行で連結。旧 `.doc` は対象外 |

- 取り出したテキストは入力欄へ。**既定は「置き換え」**。「**末尾に追加する**」チェックを用意し、ONなら追記。
- 抽出は完璧ではない（PDFの段組み・表は乱れうる）。下書きを流し込む位置づけで、ユーザーが直してから生成する前提。

## 3-3. Irodori の長文分割（チャンク）

**エンジンで扱いが正反対。** Irodori＝約30秒上限で**分割必須**、Qwen＝10分超でも安定なので**分割しない**
（むしろ繋ぎ目で品質が落ちる）。**分割は Irodori アダプタの中だけに閉じ込める。**

### 分割ルール（Irodori アダプタ内）

1. テキストを「。」「！」「？」で文に区切る。
2. 文を順に足し、**累計が100文字以上になったらそこで1チャンク確定**（文の途中では切らない＝「100文字を超えた最初の文末で切る」）。
3. **保険**：1文だけで120〜150文字を超えるなら、その文を「、」でさらに分割。
4. 各チャンクを順に生成し、**間に0.2〜0.3秒の無音**を挟んで1本に連結。
5. 連結する音声のサンプルレートを揃える。

```
chunk = ""
chunks = []
for 文 in split_by_。！？(text):
    chunk += 文
    if len(chunk) >= 100:
        chunks.append(chunk); chunk = ""
if chunk:
    chunks.append(chunk)
# （各 chunk が長すぎる場合は「、」で再分割する保険）
```

- 閾値（100文字）・無音（0.25秒）は実機で崩れ始める所を見て微調整してよい。
- **Qwen アダプタには分割を入れない**（そのまま渡す）。

## 3-4. 音量・ピッチの調整（→ 後処理）

- **音量（volume）**：線形ゲイン（`samples * gain`）。最後に **[-1.0, 1.0] にクリップ**して歪み防止。
  UIは%表示（100%=等倍）、範囲例 0〜200%。
- **ピッチ（pitch）**：半音単位で `librosa.effects.pitch_shift`（長さは変えない）。
  UIは −12〜+12 半音、初期0。0のときは処理スキップ。
- UI：既存「速度」スライダーの並びに「音量」「ピッチ」を追加。各右に現在値表示。

## 3-5. 読み上げ位置のハイライト（🟦 アプリ側）— 文単位のMVP

単語レベル（強制アラインメント）は重いので、**文単位＋文字数按分**のシンプル方式。両エンジン共通。

1. 表示テキストを「。」「！」「？」で文に分け、各文の文字インデックス（開始・終了）を覚える。
2. 完成音声の総再生時間 `D` を、各文の**文字数の比**で按分してタイムライン化（文 i は `t_i〜t_{i+1}`）。
3. 再生中、**50〜100msごとのUIタイマー**（`after`）で現在の再生位置を取り、含まれる文だけ背景色を付ける（前は消す）。
4. 停止/一時停止で止める。テキスト編集でハイライト解除（3-1のロックと連動）。
- 再生位置は、位置を返せるプレーヤー（`pygame.mixer` 等）か、再生開始からの経過時間で代用。
- `CTkTextbox` は内部 tkinter Text のタグ（`tag_config`/`tag_add`/`tag_remove`）が使える。
- 文字数按分なので多少ずれる（漢字/かな、句読点の間）。文単位なら実用上は許容。

## 3-6. 進捗表示（分割時のみ）

**正確な進捗（◯/◯）を出せるのは分割生成のときだけ**（チャンク数を事前に知っているから）。

- 進捗バー・件数表示は、**2つ以上のチャンクに分割されたとき（＝Irodoriの長文分割が走ったとき）だけ**出す。
- 次は進捗バーを出さず「生成中…」だけにする：
  - Qwenでの生成（分割しない）
  - Irodoriでもチャンクが1つの短文
- 分割時は `progress_callback(i, n)` で「完了チャンク数/全チャンク数」を更新。
  別スレッドから呼ばれるので、UI反映はメインスレッド（`after` 等）で。

## 3-7. 設定の保存（最小限）

- **保存する項目（`settings.json`）：エンジン（Qwen/Irodori）／声（話者）／表示言語／音声言語。**
  - ※ 当初は「エンジンと声の2つだけ」の方針だったが、その後「表示言語」「音声言語」も保存対象に追加した。
- **ウィンドウの位置・サイズ**は別ファイル `window_state.json` に保存（マルチディスプレイ対応で、全モニタを合わせた仮想デスクトップ範囲にクランプ）。
- **保存しない項目**：速度・音量・ピッチ（毎回デフォルト=1.0倍/100%/0半音から開始）、入力テキスト。
- アプリ終了時に保存し、次回起動時に復元。
- 復元時のフォールバック（落とさない）：保存された声がそのエンジンに無ければ既定の声へ、音声言語がそのエンジンで使えなければ既定言語へ、表示言語が対象外ならOS言語（さらに対象外なら英語）へ。

## 3-8. UIへの足し方（参考）

| 機能 | 置き場所 | 部品 |
|---|---|---|
| 生成中ロック | 既存の入力欄／生成ボタン | 部品追加なし（生成中 `disabled`） |
| ファイル読み込み | 入力欄の上 or 近く | 「ファイルを開く」ボタン＋「末尾に追加する」チェック |
| Irodori分割 | 裏側 | 進捗だけ表示（3-6） |
| 音量・ピッチ | 既存「速度」の並び | スライダー2本（現在値表示つき） |
| ハイライト | 既存入力欄 | 部品追加なし（文に背景色タグ） |

- 速度・音量・ピッチは**横1行に並べる**。ウィンドウ幅が狭く3つが入りきらないときは**縦並びに切り替える**
  （リサイズを監視し、閾値より狭ければ縦3段、広ければ横1行）。

# ■ フェーズ4：声を作る／喋り方の指定

声の指定方法を増やし、喋り方（感情・スタイル）も指定できるようにする。

## 4-0. 足すもの

1. **タブの並び替え**：「プリセット ／ 声を作る ／ 保存した声」
2. **「声を作る（VoiceDesign）」タブ新設**：参照音声なしで、文字列だけで声を作る
3. **「喋り方（感情・スタイル）」欄**：プリセットと保存した声に追加

## 4-1. 一番大事な前提：エンジンで声の作り方が違う

| | プリセット | 声を作る | 保存した声(クローン) | 喋り方(感情) |
|---|---|---|---|---|
| **Qwen3** | あり（名前付き話者） | あり（instruct） | あり | 別引数 `instruct`（声と分離可） |
| **Irodori** | モデルには名前付き話者は**無い**（アプリ側でデフォルト＋説明文ベースの声を用意して選択可） | あり（caption・得意） | あり | caption に同居 ＋ 本文の絵文字（分離不可） |

→ Irodori は「声を作る」と「感情をつける」が同じ caption の中で一体。独立した感情指定ができない。
　（このため UI では、声の選択は両エンジン可・**喋り方欄だけ Irodori でグレーアウト**としている。）

## 4-2. UIの方針（タブと出し分け）

```
[ プリセット ] [ 声を作る(VoiceDesign) ] [ 保存した声 ]
```

- **プリセットタブ**：声のプルダウンは**両エンジンで有効**。
  - Qwen … 名前付き話者（おの あんな 等）。
  - Irodori … モデルに名前付き話者は無いが、アプリ側で「デフォルト（指定なし）」＋説明文ベースの声（calm_female / bright_male / gentle_neutral の3種）を用意し、選べるようにしている。
  - ※ 当初設計は「プリセットタブはQwen専用、Irodori時は中身を全部グレーアウト」だったが、Irodoriでも声を選べた方が便利なため、**声プルダウンは両対応に変更**した（グレーアウトするのは下記の喋り方欄のみ）。
- **声を作るタブ**：**Qwen / Irodori 両対応**。声の説明文の入力欄を置く。
- **保存した声タブ**：**両対応**（どちらもクローン可）。既存の `.mvsvoice` をそのまま使う。
- **喋り方欄**（プリセット・保存した声タブに追加）：
  - **Qwen**：有効。文字列を `instruct` として渡す。
  - **Irodori**：**グレーアウト**（state無効＋背景グレー）。感情は「声を作る」の説明文に書くか、本文に絵文字を入れる運用。
- エンジン切替時は即座に有効/無効が切り替わる。

> 「UIは共通、裏でエンジンの作法に振り分ける」＝アダプタ方式そのもの。
> 使えない組み合わせはグレーアウトで明示する。

## 4-3. 「声を作る（VoiceDesign）」タブ

- UI：声の説明（複数行）。例「落ち着いた20代女性の、温かみのある声」。生成ボタンは共用でよい（タブが「声を作る」なら説明文を使う分岐）。
- アダプタへの渡し方：
  - **Qwen**：`generate_voice_design(text=本文, instruct=説明文, language=...)`。**VoiceDesignは1.7Bのみ対応**（0.6Bは非対応）。VoiceDesign用モデルを読み込む。
  - **Irodori**：VoiceDesign版モデル（caption対応）で `caption=説明文`。**通常版とは別モデル**を読み込む点に注意。
- 注意（モデルカードの実情）：
  - **声が毎回少しブレる**（話者を固定する仕組みが無い。Qwen・Irodori共通）。
    対策：気に入った声が出たら**参照音声として保存（＝保存した声に登録）**して使い回す（自動固定の仕組みは将来対応）。
  - 説明はシンプルに（矛盾指示は不安定）。Irodoriは日本語のみ・漢字弱い点は従来どおり。

## 4-4. 「喋り方（感情・スタイル）」欄

- UI：プリセット・保存した声タブに1行の入力欄。Irodori選択時はグレーアウト。
- 渡し方：
  - **Qwen / プリセット**：`generate_custom_voice(text, speaker, instruct=喋り方, language)`
  - **Qwen / 保存した声（クローン）**：喋り方欄は表示し `synthesize_clone(..., style=...)` に渡るが、
    **現状クローンには適用されない**。Qwen の `generate_voice_clone` が `instruct`（喋り方）を受け取らないため
    （声の再現が優先される API のため）。UI上は欄を出しつつ、未適用である旨をコードのコメントに残している。
  - **Irodori**：このフェーズでは渡さない（グレーアウト）。

## 6. 進め方（Claude Code への共有事項）

- 依頼者はコードを読める。実装の意図や設計判断を、要点を日本語で説明しながら進めてほしい。
- 一度に全部進めず、各フェーズの「作る順番」のステップごとに区切って、動作を確認しながら進めてほしい。
  正常に進む限りは確認待ちで止まらず、自分で確かめて次へ進んでよい。詰まったときだけ止まって原因を説明する。
- **UIは CustomTkinter のデスクトップアプリ。ブラウザ（Web/Gradio）ではない。**
- **アダプタ方式・別スレッド生成**を全フェーズで維持。
- **音量・ピッチ・速度は共通層の後処理に集約**、**Irodori分割は Irodori アダプタの中に閉じ込める**。
- **エンジンによる出し分け**（Qwen=感情を別指定可／Irodori=感情はcaption同居で独立指定不可）を、
  UIのグレーアウトとアダプタ内の振り分けの**両方**で正しく扱う。
  声の選択自体は両エンジンで可能とし、Irodori では「喋り方」欄のみグレーアウトする。
- まずフェーズ1から順に、未実装のフェーズへ進むこと。
