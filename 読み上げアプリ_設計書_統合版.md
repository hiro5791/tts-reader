# 読み上げアプリ 設計書（統合版）

> この文書は、アプリ **Multi Voice Studio**（現行 v1.0.1.0）の設計と実装の対応表です。
> もともとは「これから作る指示書」でしたが、フェーズ1〜4はすべて実装済みで、
> PyInstaller + MSIX による Microsoft Store 配布まで到達しています。
> そのため本書は「何を作るか」ではなく「**実際にどう作られているか**」を、

## 0. このアプリは何か（ゴール）

自分のPCで動く、多言語テキストの読み上げ **Windowsデスクトップアプリ**（名称：Multi Voice Studio）。

テキストを入力すると音声で読み上げ、声・速度・音量・ピッチを選べる。ファイルから読み込め、長文も扱え、
言葉で声を作ったり喋り方を指定したりできる。**Qwen3-TTS** と **Irodori-TTS** の2エンジンを切り替えて使う。

実装は「動く最小の核」から機能を足す形で進め、現在は下記すべてが動いている：

- **フェーズ1（最小版）**：テキスト→音声→再生。Qwen3 と Irodori を切り替えて聞き比べる。 ✅
- **フェーズ2（必須機能）**：話者（声）の選択／読み上げ速度の調整。 ✅
- **フェーズ3（標準機能）**：生成中ロック／ファイル読み込み／Irodori長文分割／音量・ピッチ／ハイライト／進捗表示／設定保存。 ✅
- **フェーズ4（声を作る・喋り方）**：VoiceDesign（言葉で声を作る）／感情・スタイル指定。 ✅
- **配布**：PyInstaller で1つにまとめ、MSIX で Microsoft Store 提出（v1.0.1.0）。 ✅

### 設計時から増えた実装（本書に反映済み）

当初の設計書に無かった／曖昧だったが、実装で確定・追加した主なもの：

- **多言語UI（表示言語20言語）**：`i18n.py` に翻訳辞書を持ち、OSのロケールから初期言語を自動判定。
- **波形表示＋再生位置カーソル**（matplotlib）。
- **4状態のUIステートマシン**（idle / generating / ready / playing）。
- **VRAM管理**：モードごとに別モデル（別チェックポイント）を読むため、使わないモデルを解放して同時に載るのを実質1つに保つ（8GB級GPUのクラッシュ対策）。
- **GPU/CPU自動フォールバック**（`tts/device.py`）。
- **書き込み可能データフォルダの一元管理**（`tts/paths.py`。配布版はインストール先が読み取り専用のため）。
- **クラッシュ記録**（faulthandler → `logs/crash.log`）。
- **mp3書き出し**（wav と同時保存）／**声の保存**（`.mvsvoice`）。
- **メニューバー**（ヘルプ＞バージョン情報／お問い合わせ＞問題を報告）。

## 1. 使うエンジン（重要：ライセンス）

- **Qwen3-TTS** … Apache 2.0（商用利用OK）。多言語対応（日本語含む）。プリセット話者あり。
- **Irodori-TTS** … 日本語特化。コードは MIT。
  - **日本語のみ対応**。**漢字の読みが弱め**で、難しい漢字は誤読することがある。

どちらも商用でクリーンに使えることが採用理由。
**ライセンスは「モデルの重み」の条件がコードと別管理**のことがあるため、
商用化の前に各モデルカードの最新ライセンスを必ず確認すること。

### 実際に使っているモデル（実装準拠）

| 用途 | エンジン | HuggingFace モデルID | 精度 |
|---|---|---|---|
| プリセット声 | Qwen | `Qwen/Qwen3-TTS-12Hz-1.7B-CustomVoice` | GPU=fp16 / CPU=fp32 |
| 声を作る(VoiceDesign) | Qwen | `Qwen/Qwen3-TTS-12Hz-1.7B-VoiceDesign` | GPU=fp16 / CPU=fp32（1.7Bのみ対応） |
| 保存した声(クローン) | Qwen | `Qwen/Qwen3-TTS-12Hz-1.7B-Base` | **float32 必須**（重い→CPU落ちしやすい） |
| 基本／クローン | Irodori | `Aratako/Irodori-TTS-500M-v3` | fp32 |
| 声を作る(VoiceDesign) | Irodori | `Aratako/Irodori-TTS-600M-v3-VoiceDesign` | fp32 |
| コーデック（Irodori共通） | Irodori | `Aratako/Semantic-DACVAE-Japanese-32dim` | fp32 |

> Qwen 側は「プリセット」「声を作る」「クローン」で**別チェックポイント**を読む。
> どれも数GBあり、同時に載せると 8GB 級GPUで VRAM が枯渇するため、
> **切り替え時に他モデルを解放して実質1モデルに保つ**（§3のVRAM管理）。

### エンジンの性質（設計に直結する重要な違い）

| | Qwen3-TTS | Irodori-TTS |
|---|---|---|
| 対応言語 | 10言語（日本語含む） | 日本語のみ |
| プリセット話者 | **あり**（9種：おの あんな 他） | モデルには**無い**（アプリ側でデフォルト＋説明文ベースの声を用意） |
| 長文 | **10分超でも安定**（分割不要） | **約30秒で頭打ち**（分割が必須） |
| 声を作る(VoiceDesign) | あり（`instruct`） | あり（`caption`／むしろ得意） |
| クローン | あり（Baseモデル） | あり（`--ref-wav`） |
| 感情・スタイル | 別引数 `instruct`（声と分離可） | caption に同居 ＋ 本文の絵文字（分離不可） |
| 漢字の読み | 比較的安定 | 弱い |

> この「プリセットの有無」「長文の扱い」「感情指定の作法」の3点が、UIの出し分けに効いている。

## 2. 動作環境

- 実行機：Windows。**NVIDIA GPU があれば高速**、無ければ CPU で動く（遅いが落ちない。`tts/device.py` が自動判定）。
- **Windowsの注意**：Qwen3 は FlashAttention2 が Windows非対応のため、
  全モデルで `attn_implementation="sdpa"` を使う（どの環境でも動く）。
- **8GB級GPU**：クローン（Base・float32・約6.8GB）は載りきらないことが多く、その場合は自動でCPU実行。
  プリセット／VoiceDesign（1.7B・fp16・約3.4GB）はGPUに載る。

## 3. 全体の設計方針（アダプタ方式）

「アプリ本体（UI）」と「エンジン部分」を分けて作る。**この構造は全フェーズで維持している。**

```
[ UI: app.py（CustomTkinter デスクトップ / 多言語 i18n）]
        ↓ 文章・エンジン名・声・速度・音量・ピッチ・モード等を渡す
[ tts/adapter.py（共通の窓口 synthesize / synthesize_clone ＋ 後処理DSP ＋ VRAM管理）]
        ↓ engine と mode で振り分け
   ┌───────────────┬──────────────────┬───────────────┬──────────────────┐
[qwen_engine]   [qwen_design_engine]   [clone_engine]   [irodori_engine]
 プリセット        声を作る(VoiceDesign)   クローン(Base)     Irodoriの全モード
```

補助モジュール：`audio_utils.py`（後処理DSP・無音連結）／`device.py`（GPU/CPU選択）／
`paths.py`（書き込み可能フォルダ）／`player.py`（再生・停止）。

- **UI** はエンジンの細かい違いを知らない。アダプタに「この文章を、このエンジン／モードで」と頼むだけ。
- **`tts/adapter.py`** が共通の窓口。`synthesize(...)` と `synthesize_clone(...)` の2関数を持つ。
- **音量・ピッチ・速度などの後処理（DSP）は共通層 `audio_utils.apply_dsp` に集約**。各エンジンは「生の音声を返す」ことに集中する。
- **Irodoriの長文分割は `irodori_engine.py` の中だけに閉じ込める**（共通層からは「長文でも1回で返る」ように見える）。
- **VRAM管理も共通層に集約**：`adapter._free_models(keep=...)` が、これから読むモデル以外を解放してから各エンジンを呼ぶ。

### 共通の窓口（実装の署名）

`tts/adapter.py` より（`tts/__init__.py` から `synthesize` / `synthesize_clone` / `list_voices` /
`default_voice` / `list_languages` / `default_language` / `ENGINES` を公開）：

```python
synthesize(
    text: str,
    engine: str,                 # "qwen3" | "irodori"
    voice: str | None = None,    # プリセットの声ID（None なら既定）。mode="preset" で使う
    speed: float = 1.0,          # 1.0=等倍（共通後処理）
    volume: float = 1.0,         # 1.0=等倍（共通後処理）
    pitch: float = 0.0,          # 0=変化なし。半音単位（共通後処理）
    mode: str = "preset",        # "preset" | "voice_design"
    voice_description: str | None = None,  # 声を作る説明文（Qwen=instruct / Irodori=caption）
    style: str | None = None,    # 喋り方（Qwen プリセットの instruct。Irodori は未使用）
    language: str | None = None, # 音声言語（None なら各エンジンの既定言語）
    progress_callback=None,      # 分割生成の進捗通知 callback(i, n)（Irodori 長文）
    cancel_event=None,           # 生成の中断用（停止ボタンから渡す threading.Event）
) -> str  # 生成した wav ファイルのパス
```

- **「保存した声（クローン）」は別の窓口** `synthesize_clone(...)`（`mode="saved"` ではない）：

```python
synthesize_clone(
    engine, text, ref_wav, ref_text=None, language=None,
    speed=1.0, volume=1.0, pitch=0.0,
    style=None,                  # UIからは渡るが現状クローンには未適用（後述）
    progress_callback=None, cancel_event=None,
) -> str
```

### VRAM管理（`adapter._free_models`）

各エンジンは読み込んだモデルを保持し続けるため、プリセット→声を作る→クローン…と
切り替えると VRAM が累積し、8GB級GPUで枯渇してクラッシュ（`c10.dll` アクセス違反）する。
そこで **新しいモデルを読む前に、今使わないモデルを解放**して、同時に載るのを実質1モデルに保つ：

- 対象：`tts.qwen_engine` / `tts.qwen_design_engine` / `tts.clone_engine`（`keep=` で1つだけ残す）。
- Irodori は都度ランタイムを作り、生成後に `clear_cached_runtime()` で明示解放する。
- 解放後は `gc.collect()` ＋ `torch.cuda.empty_cache()`。

# ■ フェーズ1：最小版（実装済み）

- テキスト入力→ボタン→音声で読み上げ→再生。エンジンは Qwen3 / Irodori を切替。
- UI は **CustomTkinter のデスクトップアプリ（`app.py`）**。ブラウザ版ではない。
  - ※ 予備として旧ブラウザ版 `app_web.py`（Gradio）も残してある（`python app_web.py` で起動する参考用）。
- **重い生成は別スレッド**で実行し、結果は**キュー経由でメインスレッド**が受け取ってUIを更新する
  （Tkinter は別スレッドから触ると不安定なため）。

### エンジンの呼び出し方（実装メモ）

**Qwen3-TTS**（`tts/qwen_engine.py`）：`qwen-tts` パッケージを import。
`Qwen3TTSModel.from_pretrained(..., attn_implementation="sdpa")` で読み込み、
`model.generate_custom_voice(text, language, speaker, instruct)` で生成。

**Irodori-TTS**（`tts/irodori_engine.py`）：2経路を自動で使い分ける。
- **in-process**（配布版／依存が揃った環境）：`vendor/Irodori-TTS` を import 経路に足し、
  `irodori_tts.inference_runtime` を直接呼ぶ。uv も別 venv も不要。
- **uv サブプロセス**（開発時フォールバック）：`uv run python infer.py --hf-checkpoint ... --text ...` を実行。
  依存が同一環境に揃っていない場合に自動でこちらへ切り替わる。

# ■ フェーズ2：必須機能（声選択・速度）（実装済み）

## 2-1. 話者（声）の選択

- **Qwen（CustomVoice）**：プリセット話者9種を `speaker` で切り替える。
  `ono_anna`（初期値）/ `serena` / `vivian` / `sohee` / `aiden` / `ryan` / `dylan` / `eric` / `uncle_fu`。
- **Irodori**：モデルにプリセット話者は無い。アプリ側で「デフォルト（`--no-ref`）」＋
  説明文ベースの声3種（`calm_female` / `bright_male` / `gentle_neutral`）を用意して選べるようにしている。
- 共通窓口の `list_voices(engine)` が `(表示名の i18n キー, 声ID)` を返し、
  エンジンを切り替えると声のプルダウンも入れ替わる。表示名は表示言語に追従する。

## 2-2. 読み上げ速度の調整

- **Qwen**：`generate_custom_voice` に速度パラメータが無いため、**後処理のタイムストレッチ**（ピッチ保持）で実現。
- **Irodori**：`--duration-scale`（速度の逆数）で本体側に指定できるが、
  実装では**両エンジンとも共通後処理で速度を効かせる**方針に統一している
  （分割チャンクは等倍で作り、連結後にまとめて後処理）。
- UIはスライダー（0.5〜2.0、初期1.0）。後処理は `librosa.effects.time_stretch`。

# ■ フェーズ3：標準機能セット（実装済み）

## 3-0. 後処理（DSP）を1か所に集約 — `tts/audio_utils.py`

両エンジンとも音量・ピッチのつまみを持たないので、**生成後の後処理**で実現。`apply_dsp` に集約し、処理順は：

```
[ エンジンが生の音声を作る ]
   → ① ピッチ変更（pitch != 0）   librosa.effects.pitch_shift（長さ不変・半音単位）
   → ② 速度変更（speed != 1.0）   librosa.effects.time_stretch（高さ不変）
   → ③ 音量変更（volume != 1.0）  線形ゲイン → 最後に [-1,1] にクリップ
   → [ 完成した音声を返す ]
```

- 速度=1・ピッチ=0・音量=1 のときは**素通し**（無駄な読み書き・劣化を避ける）。
- `adapter._postprocess` が、後処理が要るときだけ別ファイル（`*_dsp.wav`）に書き出してそのパスを返す。

## 3-1. 生成中のテキスト編集ロック（🟦 アプリ側）

- 生成開始時にテキスト欄を `disabled`、各ボタンを状態に応じて無効化、「生成中…」表示。
- 完了/エラー時にメインスレッドのコールバックで元に戻す。UI部品の状態変更は**必ずメインスレッド**。
- これは**4状態のステートマシン**（idle / generating / ready / playing）に統合されている（§3-9）。

## 3-2. ファイル読み込み（txt / PDF / Word）（🟦 アプリ側）

- 「ファイルを開く」ボタン（`.mvsvoice` の「ファイルを選択」とは別物）。`filedialog` で開き拡張子で振り分け：

| 種類 | ライブラリ | メモ |
|---|---|---|
| `.txt` | 標準（`open`） | UTF-8で試し、失敗したら cp932（Shift-JIS）で再試行 |
| `.pdf` | `pypdf` | レイアウト崩れ・表が多いと乱れうる |
| `.docx` | `python-docx` | 段落を改行で連結。旧 `.doc` は対象外 |

- 取り出したテキストは入力欄へ。既定は「置き換え」、「末尾に追加する」チェックONで追記。

## 3-3. Irodori の長文分割（チャンク）— `irodori_engine.split_text`

Irodori＝約30秒上限で**分割必須**、Qwen＝分割しない。**分割は Irodori アダプタの中に閉じ込める。**

分割ルール（実装値）：

1. テキストを「。」「！」「？」**と改行**で文に区切る（区切り文字は文末に残す）。
2. 文を順に足し、**累計が100文字以上**（`_CHUNK_MIN_CHARS`）になったら文末でチャンク確定。
3. **保険**：1文だけで**140文字超**（`_LONG_SENTENCE_CHARS`）なら「、」「,」で更に分割してから足す。
4. 各チャンクを順に生成し、**間に0.25秒の無音**（`_CHUNK_GAP_SEC`）を挟んで1本に連結（`concat_with_silence`）。
5. 1チャンクなら連結せずそのまま返す。**Qwen には分割を入れない**。

## 3-4. 音量・ピッチの調整（→ 後処理）

- **音量**：線形ゲイン、最後に [-1,1] クリップ。UIは%表示（100%=等倍、0〜200%）。
- **ピッチ**：半音単位で `librosa.effects.pitch_shift`。UIは −12〜+12 半音、初期0（0はスキップ）。
- UI：速度・音量・ピッチを**横1行に並べる**。実装上は狭くても横1行を維持する
  （縦3段にすると背が高くなり保存ボタン等が画面外に出るため。閾値 `SLIDERS_WRAP_WIDTH=400`は最小幅600では実質発動しない）。

## 3-5. 読み上げ位置のハイライト（🟦 アプリ側）— 文単位のMVP

単語レベル（強制アラインメント）は重いので、**文単位＋文字数按分**方式。両エンジン共通。

1. 表示テキストを「。」「！」「？」で文に分け、各文の文字インデックスを覚える（`_sentence_spans_of`）。
2. 完成音声の総再生時間 `D` を、各文の**文字数比**で按分してタイムライン化（`_build_highlight_timeline`）。
3. 再生中、**UIタイマー**（`after`）で**再生開始からの経過時間**を見て、含まれる文だけ背景色（薄い黄 `#ffe9a8`）を付ける。
4. 停止/一時停止で止め、テキスト編集でハイライト解除（3-1のロックと連動）。
- 再生は `tts/player.py`（**sounddevice**）。位置は再生位置APIではなく**経過時間で代用**するため、
  波形の再生カーソルとハイライトは同じ経過時間ベースで動く。
- `CTkTextbox` 内部 tkinter Text のタグ（`tag_config`/`tag_add`/`tag_remove`）を使う。

## 3-6. 進捗表示（分割時のみ）

- **2つ以上のチャンクに分割されたとき（Irodori長文）だけ** `progress_callback(i, n)` で「i/n」を左下ステータスに表示。
- Qwen（分割しない）／Irodoriの短文（1チャンク）は「生成中…」のみ。
- callback は別スレッドから呼ばれるので、UI反映はメインスレッド（`after`）へ回す。

## 3-7. 設定の保存 — `settings.json` / `window_state.json`

- **`settings.json`（保存項目）**：エンジン／声（話者）／**表示言語（`lang`）**／**音声言語（`audio_lang`）**。
- **`window_state.json`**：ウィンドウの位置・サイズ（マルチディスプレイ対応で、全モニタを合わせた仮想デスクトップ範囲にクランプ）。
- **保存しない**：速度・音量・ピッチ（毎回 1.0倍/100%/0半音から開始）、入力テキスト。
- 保存先は**書き込み可能なデータフォルダ**（`tts/paths.data_root()`）。
  配布版は `%LOCALAPPDATA%\MultiVoiceStudio\`、開発時はプロジェクト直下。
- 復元時のフォールバック：声がそのエンジンに無ければ既定へ、音声言語が使えなければ既定言語へ、
  表示言語が対象外ならOS言語（さらに対象外なら英語）へ。

## 3-8. UIへの足し方

| 機能 | 置き場所 | 部品 |
|---|---|---|
| 生成中ロック | 既存の入力欄／各ボタン | ステートマシンで一括 `disabled` |
| ファイル読み込み | 入力欄の近く | 「ファイルを開く」ボタン＋「末尾に追加する」チェック |
| Irodori分割 | 裏側 | 進捗だけ表示（3-6） |
| 音量・ピッチ | 「速度」の並び | スライダー（現在値表示つき、横1行） |
| ハイライト | 既存入力欄 | 文に背景色タグ |
| 波形＋再生カーソル | 入力欄の下 | matplotlib キャンバス |

## 3-9. UIステートマシン（実装で確定）

画面は「状態」を1つ持ち、それに合わせてボタンの有効/無効・ステータス・波形カーソルが切り替わる：

- `idle` … 起動直後（生成だけ押せる）
- `generating` … 生成中（すべて押せない。停止で中断できる）
- `ready` … 生成完了で再生待ち（生成・再生が押せる）
- `playing` … 再生中（生成・停止が押せる）

## 3-10. その他の実装機能

- **波形表示＋再生位置カーソル**（matplotlib）：生成後に波形を描き、再生中はカーソルが進む。
- **mp3書き出し**：「音声を保存」で **wav と mp3 を同時に**書き出す（`soundfile` の `format="MP3"`）。
- **多言語UI（20言語）**：`i18n.py`。起動時にOSロケールから初期表示言語を自動判定（対象外は英語）。
  右上のメニューで切替でき、切替時に画面の文字が即座に貼り替わる。
- **メニューバー**：ヘルプ＞バージョン情報／お問い合わせ＞問題を報告（Googleフォームを開く）。
- **クラッシュ記録**：`faulthandler` でネイティブ致命傷・未捕捉例外を `logs/crash.log` に残す（配布版の無言クラッシュ対策）。

# ■ フェーズ4：声を作る／喋り方の指定（実装済み）

## 4-0. タブ構成

```
[ プリセット ] [ 声を作る(VoiceDesign) ] [ 保存した声 ]
```

内部キーは `preset` / `design` / `saved`（`_active_voice_tab()` が返す）。

## 4-1. エンジンで声の作り方が違う

| | プリセット | 声を作る | 保存した声(クローン) | 喋り方(感情) |
|---|---|---|---|---|
| **Qwen3** | あり（9話者） | あり（instruct） | あり（Base） | 別引数 `instruct`（声と分離可） |
| **Irodori** | モデルには無い（アプリ側でデフォルト＋説明文ベース3種） | あり（caption・得意） | あり（`--ref-wav`） | caption に同居（分離不可） |

→ Irodori は「声を作る」と「感情をつける」が同じ caption の中で一体。独立した感情指定ができない。
　このため UI では、**声の選択は両エンジン可・喋り方欄だけ Irodori でグレーアウト**としている。

## 4-2. UIの出し分け

- **プリセットタブ**：声のプルダウンは**両エンジンで有効**。
  - Qwen … 名前付き話者（おの あんな 等9種）。
  - Irodori … デフォルト＋説明文ベース3種（calm_female / bright_male / gentle_neutral）。
- **声を作るタブ**：**Qwen / Irodori 両対応**。声の説明文の入力欄を置く。
- **保存した声タブ**：**両対応**。`.mvsvoice`（zip）をファイル選択またはドラッグ&ドロップで指定。
- **喋り方欄**（プリセット・保存した声タブ）：Qwen=有効（`instruct`）、Irodori=**グレーアウト**。
- エンジン切替時は即座に有効/無効が切り替わる。

## 4-3. 「声を作る（VoiceDesign）」

- **Qwen**：`qwen_design_engine.synthesize_design(text, instruct=説明文, language)`。
  **1.7Bのみ対応**、`Qwen/Qwen3-TTS-12Hz-1.7B-VoiceDesign` を読み込む。
- **Irodori**：`irodori_engine.synthesize_design(text, caption=説明文)`。
  **VoiceDesign版モデル**（`Aratako/Irodori-TTS-600M-v3-VoiceDesign`）で `--caption ... --no-ref`。
- 注意：**声が毎回少しブレる**（話者固定の仕組みが無い。両エンジン共通）。
  気に入った声は「保存した声」に登録して使い回す運用（自動固定は将来対応）。

## 4-4. 「喋り方（感情・スタイル）」欄

- **Qwen / プリセット**：`generate_custom_voice(text, speaker, instruct=喋り方, language)`。
- **Qwen / 保存した声（クローン）**：欄は表示し `synthesize_clone(..., style=...)` に渡るが、
  **現状クローンには適用されない**（`generate_voice_clone` が `instruct` を受け取らないため）。
  コードにその旨のコメントを残している。
- **Irodori**：渡さない（グレーアウト）。

## 4-5. 「保存した声」ファイル形式 `.mvsvoice`

- 実体は **zip**。中身は `reference.wav`（参照音声）＋ `voice.json`（メタ情報 `format_version: 1`）。
  `voice.json` に `name` / `ref_text`（書き起こし）/ `language` / `audio_file` / `created_at` を持つ。
- 生成した音声を「声を保存」で `.mvsvoice` に書き出せる。詳細は `ボイスファイル形式_仕様_mvsvoice.md` 参照。

## 5. 配布（実装済み：PyInstaller + MSIX）

当初は「後回し」だったが、現在は Microsoft Store 提出まで到達（**v1.0.1.0**、`version.txt` で一元管理）。

- **ビルド環境**：開発用 `.venv`（`requirements.txt`）とは別に、**統合ビルド環境**（`requirements-build.txt`）を用意。
  ここでは Irodori を uv 別プロセスでなく**同一プロセス（in-process）**で動かせるよう、
  Irodori のランタイム依存（torch/torchaudio/torchcodec、peft、dacvae、silentcipher 等）を1つの環境に入れる。
  → これにより PyInstaller で1つにまとめられる（配布先に uv も別 venv も不要）。
- **PyInstaller**：`MultiVoiceStudio.spec` でビルド。`vendor/Irodori-TTS`・`version.txt` 等を同梱。
- **モデルは既定で非同梱（オプトイン）**：初回起動でダウンロード。同梱する場合は `scripts/prefetch_models.py` で
  取得したHFキャッシュを含め、frozen 時に `HF_HOME` をそこへ向けてオフライン動作させる（`app.py` 冒頭）。
- **MSIX / Store**：`packaging/`（`AppxManifest.xml`・`build_msix.ps1`・`PartnerCenter_提出手順.md`）。
  Inno Setup 版インストーラ（`MultiVoiceStudio.iss`）もある。
- **書き込み先**：配布版はインストール先が読み取り専用のため、出力wav・設定・ログは
  `%LOCALAPPDATA%\MultiVoiceStudio\`（`outputs/` `logs/` `settings.json` `window_state.json`）に書く（`tts/paths.py`）。

## 8. 設計の要点（維持すべき原則）

- **UIは CustomTkinter のデスクトップアプリ。ブラウザ（Web/Gradio）ではない**（`app_web.py` は予備）。
- **アダプタ方式・別スレッド生成**を維持。生成結果はキュー経由でメインスレッドが受け取る。
- **音量・ピッチ・速度は共通層（`audio_utils`）に集約**、**Irodori分割は `irodori_engine` に閉じ込める**。
- **VRAM管理（`_free_models`）でモデルの同時ロードを実質1つに保つ**（8GB級GPUのクラッシュ対策）。
- **エンジンによる出し分け**（Qwen=感情を別指定可／Irodori=caption同居で独立指定不可）を、
  UIのグレーアウトとアダプタ内の振り分けの**両方**で扱う。声選択は両エンジン可、Irodoriは喋り方欄のみグレーアウト。
- **配布版は書き込み先を `paths.py` 経由**にする（インストール先が読み取り専用のため）。
