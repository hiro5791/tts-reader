> **フェーズ1（最小版）→ 動作確認 → フェーズ2（必須機能の残り）** の順に進めてください。

---

## 0. このアプリは何か（ゴール）

自分のPCで動く、日本語テキストの読み上げ **Windowsデスクトップアプリ**。

最終的に目指すのは、テキストを入力すると音声で読み上げ、声や速度を選べるアプリ。
ただし一度に全部は作らず、まず「動く最小の核」を作ってから機能を足していく。

- **フェーズ1（最小版）**：テキスト→音声→再生。Qwen3 と Irodori を切り替えて聞き比べる。
- **フェーズ2（必須機能の残り）**：話者（声）の選択／読み上げ速度の調整 を足し、
  「読み上げアプリとして成立する最低ライン」を完成させる。

---

## 1. 使うエンジン（重要：ライセンス）

- **Qwen3-TTS** … Apache 2.0（商用利用OK）。0.6B または 1.7B モデル。多言語対応（日本語含む）。
- **Irodori-TTS** … 日本語特化。コードは MIT。
  - 注意：Irodori は **日本語のみ対応**。**漢字の読みが弱め**で、難しい漢字は誤読することがある
    （最小版では許容してよい。辞書補正は後の拡張で対応）。

どちらも商用でクリーンに使えることが採用理由。
**ライセンスは「モデルの重み」の条件がコードと別管理**のことがあるため、
商用化の前に各モデルカードの最新ライセンスを必ず確認すること。

---

## 2. 動作環境

- 開発機：Windows または Linux の、**NVIDIA GPU 搭載PC**（このPCで動かす前提）。
- GPUがある前提で問題ない。CPUのみだと非常に遅いので、GPUを使う構成にする。
- Python を使う（Claude Code が必要なものをインストールしてよい）。
- **Windowsの注意**：Qwen3 は FlashAttention2 が Windows非対応。
  `attn_implementation="sdpa"`（または `--no-flash-attn`）で回避する。

---

## 3. 全体の設計方針（アダプタ方式）

最小版でも、後の拡張がラクになるよう、最初から「アプリ本体（UI）」と「エンジン部分」を分けて作る。

```
[ UI（デスクトップ画面：テキスト入力＋エンジン選択＋再生ボタン） ]
        ↓ 文章・エンジン名・(後で)声・速度 を渡す
[ TTSアダプタ（共通の窓口） ]
        ↓ エンジンごとに振り分け
   ┌──────────────┬──────────────┐
[ Qwen3 アダプタ ]      [ Irodori アダプタ ]
```

- **UI** は、エンジンの細かい違いを知らない。アダプタに「この文章を、このエンジンで」と頼むだけ。
- **TTSアダプタ** が共通の窓口。`synthesize(...)` という1つの関数を持つ。
- **各エンジンのアダプタ** が、実際に Qwen3 / Irodori を呼び出して音声を作る。
- これにより、後でエンジンや機能を足したり差し替えたりするのが簡単になる。
  **この構造はフェーズ1から採用する。**

# ■ フェーズ1：最小版

## 1-1. ゴール

- テキスト入力欄に文章を貼り付ける
- ボタンを押すと、その文章を音声で読み上げて再生する
- 読み上げエンジンを **Qwen3-TTS** と **Irodori-TTS** から切り替えて、聞き比べられる

これ以外（声の選択・速度・ファイル読み込み・オリジナルの声・誤読補正など）は
**フェーズ1には入れない**。

## 1-2. 共通の窓口（インターフェース）

フェーズ1で必要なのは、実質これ1つ：

- `synthesize(text: str, engine: str) -> 音声データ（wavファイルのパス、または音声バイト列）`
  - `engine` は `"qwen3"` か `"irodori"`
  - 中で対応するアダプタを呼び、できた音声を返す

声・速度などのオプションは、フェーズ1では固定でよい（各エンジンのデフォルト＋日本語のプリセット話者）。
※ フェーズ2で `voice` と `speed` を足すので、拡張しやすい形にしておくこと。

## 1-3. UI（Windowsデスクトップアプリ）

- **ブラウザ画面（Web/Gradio）ではなく、Windows上で動くデスクトップGUIにする。**
  最終的にはダブルクリックで起動して使うアプリを目指す。
- 実装は **CustomTkinter** を基本とする（Python製で手軽・今風の見た目になり、配布もしやすい）。
  - 本格的な見た目が必要になれば、後で **PySide6 / PyQt** に乗せ替えてよい。
  - アダプタ方式なので、UIを差し替えてもエンジン部分（アダプタ層）はそのまま使い回せる。
- 画面に必要なもの：
  1. テキスト入力欄（複数行）
  2. エンジン選択（Qwen3 / Irodori を切り替えるラジオボタンかプルダウン）
  3. 「読み上げる」ボタン
  4. 音声の再生（再生／停止ボタン。OS標準の再生機能やライブラリで鳴らす）
  5. 処理中の表示（「生成中…」など。初回はモデル読み込みで時間がかかるため）
- **重い処理は別スレッドで動かす**：音声生成中もUIが固まらないよう、生成は裏のスレッドで実行し、
  終わったらメインのUIに結果（再生可能な状態）を返す。これをしないと「生成中…」の間アプリが無反応になる。

## 1-4. エンジンの呼び出し方（参考情報）

最新の正確な使い方は各リポジトリのREADMEを確認すること。

### Qwen3-TTS

- パッケージ：`pip install -U qwen-tts soundfile`
- 公式リポジトリ：https://github.com/QwenLM/Qwen3-TTS
- 用意済み話者で読み上げる例（CustomVoiceモデル）。日本語の女性話者 `Ono_Anna` がある：

```python
import torch, soundfile as sf
from qwen_tts import Qwen3TTSModel

model = Qwen3TTSModel.from_pretrained(
    "Qwen/Qwen3-TTS-12Hz-1.7B-CustomVoice",
    device_map="cuda:0",
    dtype=torch.float16,           # GPUで安定させるため float16
    attn_implementation="sdpa",    # flash_attention_2 が使えない環境向け（Windows等）
)
wavs, sr = model.generate_custom_voice(
    text="ここに読み上げる日本語",
    language="Japanese",
    speaker="Ono_Anna",
)
sf.write("out_qwen.wav", wavs[0], sr)
```

### Irodori-TTS

- 公式リポジトリ：https://github.com/Aratako/Irodori-TTS
- OpenAI互換のAPIサーバー実装もある：https://github.com/Aratako/Irodori-TTS-Server
- コマンド（CLI）で読み上げる例（参照音声なし＝プリセット的に使う）：

```bash
python infer.py \
  --hf-checkpoint Aratako/Irodori-TTS-500M-v3 \
  --text "ここに読み上げる日本語" \
  --no-ref \
  --output-wav out_irodori.wav
```

- Irodori アダプタの実装方法は2通り。Claude Code がやりやすい方を選んでよい：
  1. `infer.py` をコマンドとして呼び出し、出力された wav を読み込む
  2. `Irodori-TTS-Server`（OpenAI互換API）を立てて、HTTPで呼び出す（将来の差し替えがしやすい）

# ■ フェーズ2：必須機能の残り

> **フェーズ1（最小版）が動いたあとに着手する。**

## 2-0. ゴール

最小版に、必須レベルで残っている2つを足して「読み上げアプリとして成立する最低ライン」を完成させる。

1. 話者（声）の選択
2. 読み上げ速度の調整

## 2-1. 話者（声）の選択

ユーザーが、読み上げる声を選べるようにする。

### エンジンごとの対応

- **Qwen3-TTS（CustomVoice）**：プリセット話者から選ぶ。
  - `generate_custom_voice(text=..., language="Japanese", speaker=話者名)` の `speaker` を切り替える。
  - 日本語話者の例：`Ono_Anna`。利用可能な話者一覧は最新のモデルカード／READMEで確認し、
    日本語で使えるものを UI のプルダウンに並べる。
- **Irodori-TTS**：プリセット話者という概念が薄い。最小実装としては、
  - まず「デフォルトの声（参照なし `--no-ref`）」を1つ用意し、
  - 余裕があれば、あらかじめ用意した参照音声（許諾済み）を「声プリセット」として何種類か登録して選べるようにする。
  - VoiceDesign（caption で声を設計）まではこのフェーズでは踏み込まない（後フェーズで対応）。

### アダプタへの追加

- 共通インターフェースを拡張する。`synthesize` に `voice` を足す：
  - `synthesize(text: str, engine: str, voice: str | None = None) -> 音声`
- `list_voices(engine) -> [話者名, ...]` を追加し、UI がエンジンを選んだとき、
  そのエンジンで選べる声の一覧を出せるようにする。
- エンジンごとに「選べる声」が違うので、エンジンを切り替えたら声のプルダウンも入れ替わるようにする。

### UI への追加

- エンジン選択の隣に「声」のプルダウンを追加。
- エンジンを変えたら、声のプルダウンの中身も自動で切り替わる。

## 2-2. 読み上げ速度の調整

ユーザーが、読み上げの速さを変えられるようにする（例：0.5倍〜2.0倍）。

### 実装方針

- まず各エンジンに速度パラメータがあるか確認する。
  - **あれば**：エンジン側のパラメータで速度を変える（音質が自然なため優先）。
  - **無ければ**：生成後の音声を後処理で伸縮する（フォールバック）。
    - ピッチを保ったまま速度だけ変える方法（タイムストレッチ）を使う。
      例：`librosa.effects.time_stretch`、`pyrubberband`、`soundstretch`（SoX）など、
      環境に入れやすいものを Claude Code が選ぶ。
    - 単純なリサンプリング（再生レート変更）だと声が高く/低くなってしまうので、
      ピッチ保持のタイムストレッチを使うこと。

### アダプタへの追加

- 共通インターフェースに `speed` を足す：
  - `synthesize(text: str, engine: str, voice: str | None = None, speed: float = 1.0) -> 音声`
- `speed=1.0` を等倍とし、各アダプタが「エンジンの速度パラメータ」または「後処理タイムストレッチ」で実現する。
  どちらで実現したかは、アダプタの内部に隠蔽してよい（UIは `speed` を渡すだけ）。

### UI への追加

- スライダー（例：0.5〜2.0、初期値1.0）を追加。
- 値を変えて読み上げると、その速度で再生される。

## 2-3. 作る順番（フェーズ2）

1. **話者選択（Qwen3）**：まず Qwen3 のプリセット話者を切り替えられるようにする。
2. **話者選択（Irodori）**：Irodori のデフォルト声＋（任意で）参照音声プリセットを足す。
3. **速度調整**：エンジン側パラメータ → 無ければ後処理タイムストレッチで実装。
4. **UI 統合**：エンジン・声・速度の3つを選んで読み上げ→再生できることを確認する。
5. ここまでで **「必須機能ぜんぶ入り」完成**。
